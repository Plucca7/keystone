# Pilar de Fluxo de trabalho — regra detalhada

> Regra por extenso do pilar de Fluxo de trabalho do Keystone. Esqueleto em [pilares.md](pilares.md).
> Colaboração aberta em [projeto-aberto](README.md). 🔧 = guarda automático.

## Princípio

O "como se trabalha" vem montado de série, pra o projeto nascer organizado e nunca perder o fio entre
sessões. O mesmo sistema que permite abrir o projeto pra comunidade sem virar bagunça.

## 1. Três níveis

- **Oficial** (o que está no ar / pronto), **preparação** (o que está sendo integrado), **trabalho
  do dia** (a frente de cada um). Nunca se mexe direto no oficial.
- Cada frente independente tem sua própria área de trabalho.

## 2. Porta de revisão antes do oficial

- Toda mudança passa por uma **conferência** antes de entrar no oficial. Mesmo trabalhando sozinho, a
  porta existe — e os guardas automáticos conferem junto.
- É a mesma porta que recebe as sugestões da comunidade (projeto aberto): a proposta passa pelos
  guardas e só o aprovado entra. Ver [README.md](README.md).

## 3. Quadro de tarefas de série

- O projeto nasce com um **quadro** (a fazer / fazendo / feito) montado. Dá pra ver o andamento desde
  o dia 1.

## 4. Passagem de bastão entre sessões

- **Diário** do que foi feito + **briefing** pra retomar. Cada sessão deixa o caminho pronto pra
  próxima; não se perde contexto.

## 5. Autoria em cada entrega

- Toda entrega registra **quem fez** (pessoa ou assistente de IA). Importante porque vários podem
  trabalhar no mesmo projeto. Conecta com o rastro do Banco.

## 6. Teste a cada entrega

- A bateria de conferência roda a cada entrega; a suíte cresce. Ver [testes.md](testes.md).
- 🔧 Guardas (qualidade + testes) na porta de revisão.
